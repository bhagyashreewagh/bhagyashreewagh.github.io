The problem

This was built at the UW eScience Institute in the context of NAIRR, the National AI Research Resource, a federally funded initiative giving researchers across universities access to large-scale AI compute. For federal accountability and cost auditing, we needed full observability into how models were being used.

Our team was running LiteLLM as an internal LLM gateway, routing requests across GPT-4, Claude, and other models on behalf of different users and teams. The problem was zero visibility. We had no idea which models were being hit the most, which users were driving cost, whether latency was degrading, or if there were anomalous usage spikes. We were manually exporting logs and graphing them in Python when we wanted to check anything.

The goal: build a real-time observability pipeline where every usage event is captured the moment it happens, processed and stored at two different time horizons, and surfaced on a live dashboard.

Architecture decision: push vs. poll

The first design question was whether to poll LiteLLM for data or have data pushed to us the moment it's available. Polling always means you're at least one interval behind. We went push-based using LiteLLM's built-in callback system. That fires on every API call completion and returns a structured payload: model name, user ID, session ID, input and output token counts, latency, cost estimate, and full request metadata.

Streaming layer: Kafka

From the LiteLLM callback, events are published directly into Kafka, which we self-hosted on Kubernetes. The topic is llm.usage.raw, partitioned by user ID so events from the same user always land on the same partition and stay in order.

The reason for introducing Kafka rather than writing directly from the callback into a database is durability and decoupling. The LiteLLM gateway never blocks waiting for anything. It fires the callback asynchronously and moves on. That's the key throughput advantage, especially under traffic spikes when you most need observability to be working.

Processing layer: Kafka consumer

A Kafka consumer group (a containerized Python script) reads from llm.usage.raw in real time. Consumer instances don't scale automatically on their own, so we used KEDA (Kubernetes Event Driven Autoscaler). It watches consumer group lag and automatically spins up more instances when lag grows, then scales back down when it clears.

Each consumer does three things in sequence: normalizes field names across model providers, computes derived fields like cost per token and session duration, and fans the processed event out to two destinations: Redis for the hot real-time layer and Azure Data Lake Storage for long-term archival. Both writes happen before committing the Kafka offset, so we never acknowledge an event as processed until it's safely in both stores.

Hot layer: Redis

Redis handles the real-time dashboard and alerting layer because we need very fast lookups by user, model, and time window. We use Redis sorted sets to store running cost and token totals by user ID and model name, making it trivial to answer questions like "how much did this user spend in the last hour?" without scanning a full table. Redis hashes store the latest state of each active session for live dashboard display.

A lightweight alerting service checks hourly cost totals every 60 seconds. If a user crosses the hourly cost threshold, it sends an alert to Slack and email via webhook. Redis fits better than a relational database here because the data structures map directly to the queries the dashboard and alerting service need.

Cold layer: data lake and historical analysis

Every raw event also lands in Azure Data Lake Storage as the archival record, the single source of truth for the full history of gateway usage. Events are written as partitioned Parquet files organized by date and model, which makes Synapse queries efficient without needing to scan the entire dataset. Azure Synapse Analytics queries the lake directly via SQL without moving data into a separate store.

Dashboard

The Streamlit dashboard is deployed on Azure Container Apps and pulls from two sources: Redis for live metrics refreshed every few seconds, and Synapse on demand for historical trend queries. It shows cost per model, cost per user, request volume over time, average latency by model, session length distribution, error rates, and token efficiency ratios across 13,000+ research sessions.

Azure Container Apps sits behind Azure's built-in load balancer automatically once you configure the ingress. The ingress is set to external with traffic routed to Streamlit's port, and the platform handles distribution from there. The full infrastructure including the Container Apps environment, ingress config, and scaling rules is defined in pulumi/__main__.py in the repo and provisioned with a single pulumi up.
