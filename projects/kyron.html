What it is

Kyron Medical is an AI front desk for a medical practice. Patients chat with Kyra, an AI assistant that collects their info, figures out which specialist they need, shows real appointment slots, and books one. After booking, it sends a confirmation email, creates a calendar event, and optionally texts a reminder. If the patient wants to switch to a phone call mid-conversation, Kyra picks up the phone already knowing everything from the chat.

The stack is React and Node.js, both in TypeScript. Claude Sonnet powers the AI. Vapi handles voice calls. Twilio handles SMS. Nodemailer sends the emails. It was originally deployed on AWS EC2 with Nginx, PM2, and Let's Encrypt. I terminated that instance to avoid ongoing costs and moved it to Render, where it's live now. The AWS deploy script is still in the repo if you want to self-host it.

The conversation workflow

The entire booking workflow is encoded in Claude's system prompt as an 11-step sequence: greet and ask for the patient's full name, then date of birth, then phone, then email, then reason for visit. Based on the reason, Kyra routes to the right specialist. Each doctor has a keyword list in the data file; back pain goes to Orthopedics, chest pain goes to Cardiology, and so on. Then Kyra presents five available slots, confirms everything, asks about SMS consent, and closes with specialty-specific prep tips.

The system prompt also has all five doctors, their office hours, and the next 30 days of available slots injected at request time. Those slots are generated deterministically using a seed-based hash so availability is consistent across requests without needing a real database.

Streaming with Server-Sent Events

When a patient sends a message, the backend opens a persistent HTTP connection and streams Claude's response back in chunks as they arrive. The frontend appends each chunk to the message bubble as it comes in, which is why the text appears to type itself. This is Server-Sent Events: one-directional, server to browser, over a long-lived connection. The patient sends one request, the server keeps the connection open and pushes tokens through it until the response is done.

The booking signal mechanism

The most interesting part of this system is how the booking actually gets triggered. Claude is a text model. It has no way to call a function or fire a webhook directly. But the system prompt instructs it that when all booking details are confirmed, it must append a structured signal to the end of its response:

APPOINTMENT_CONFIRMED:{"patientFirstName":"Jane","doctorId":"dr-chen","date":"2026-06-20","time":"2:00 PM","slotId":"dr-chen-2026-06-20-T3","smsOptIn":true,...}

The backend watches the stream token by token. The moment it sees APPOINTMENT_CONFIRMED:, it stops forwarding any further chunks to the browser. The patient never sees the raw JSON. The backend collects the full signal, parses the payload, checks that the slot hasn't already been booked by someone else, then fires everything in parallel: marks the slot taken, saves the patient record, sends the confirmation email, creates the Google Calendar event, and sends the SMS if the patient opted in. The clean response without the signal is what gets sent to the frontend.

The LLM becomes an event emitter. It decides when to fire the event and what data to put in it. The backend just listens to the stream and reacts.

Returning patient detection

Patient records are saved to a JSON file on disk after every booking. When a new chat session starts, the backend extracts any phone number or email from the conversation and looks it up in that file. If a match is found, the system prompt gets a block injected with the patient's full name, date of birth, phone, email, and last visit details, plus explicit instructions for Claude not to re-ask for any field already on file. Kyra greets the returning patient by name and skips straight to asking why they're coming in.

There's also a mid-conversation save: if a patient provides their phone and email but the session ends before booking, the system saves a stub record. So even an incomplete conversation makes the next visit smoother.

Voice handoff

When a patient clicks the call button, the backend pulls the last 10 messages from the session, strips the booking signal if present, summarizes the context, and sends it to Vapi as the voice assistant's system prompt. Vapi calls the patient's phone and Kyra picks up already knowing who they are and what they were discussing. The patient doesn't repeat themselves.

For inbound calls: Vapi sends a webhook to the backend whenever a patient calls in. The backend looks up the caller's phone in the call history records and the patient database. If there's a prior conversation on file, even from before a server restart, that context gets injected into the assistant config that Vapi receives back. The last five calls per patient are kept in memory; the patient database persists across restarts.

Integrations

Email: Nodemailer sends a styled HTML confirmation with the appointment details, doctor info, location, what to bring, and a 24-hour cancellation notice. Supports Gmail app passwords or any generic SMTP provider.

SMS: Twilio sends a reminder only if the patient explicitly said yes during the booking flow. The opt-in is captured in the booking signal JSON so the backend knows exactly what was consented to.

Google Calendar: A service account creates a calendar event on behalf of the practice and sends an invite to the patient's email. The appointment card in the chat also generates a one-click "Add to Google Calendar" link on the frontend without any backend call.

Infrastructure

It originally ran on AWS EC2 with Nginx handling incoming traffic, Let's Encrypt for HTTPS, and PM2 keeping the server process alive. I wrote a deploy script that sets the whole thing up from a blank Ubuntu server in one command. The instance is terminated now to avoid costs, and it's on Render instead. The full AWS setup is still in the repo if you want to self-host it.
